## 1. 背景
- **问题场景**: 很多 LLM 应用效果不稳定，不是模型不行，而是输入给模型的指令、上下文和约束设计得不清楚。
- **学习目标**: 建立 Prompt 设计的基础框架，理解任务说明、上下文、约束和输出格式的重要性。
- **前置知识**: 了解 LLM 基本能力、上下文窗口和简单的调用方式。

## 2. 核心结论
- 好的 Prompt 不是“越长越好”，而是任务、上下文、约束和输出格式越清晰越好。
- Prompt 的目标是减少模型猜测空间，让模型更稳定地完成指定任务。
- 输出格式要求写清楚后，结果通常会更可控、更容易接入程序。
- Prompt 设计应该和评测一起迭代，而不是凭感觉不断改词。

## 3. 原理拆解
- **关键概念**: 一个常见 Prompt 结构包括角色/任务、输入上下文、限制条件、输出格式和示例。
- **运行机制**: 模型会根据输入上下文和概率分布生成结果，Prompt 越清晰，歧义越少。
- **图示说明**: Prompt 设计本质上是在控制任务边界和输出预期。

## 4. 实战步骤

### 4.1 环境准备
- 依赖版本: 任意 LLM 调用环境
- 安装命令: 视具体 SDK 而定

```bash
echo "Prompt design starts from task clarity"
```

### 4.2 核心代码

```text
你是一名软件测试助手。

任务:
根据给定接口文档，输出一份接口测试检查清单。

要求:
1. 只输出 Markdown 列表
2. 必须覆盖状态码、字段结构、异常场景
3. 不要输出无关解释

输入文档:
{{api_doc}}
```

### 4.3 如何验证
- 本地运行命令: 在任意模型调用环境中测试多次输出结果。
- 预期结果: 输出格式稳定，核心检查点一致，噪声内容减少。
- 失败时重点检查: 任务定义是否模糊、上下文是否缺失、输出格式约束是否明确。

## 5. 项目实践建议
- **适用场景**: 摘要、结构化提取、问答、测试辅助、Agent 子任务拆解。
- **不适用场景**: 需求本身就没有定义清楚、输入信息极度不足的情况。
- **落地建议**: Prompt 要和业务目标、评测数据、失败案例一起维护。
- **与其他方案对比**: 与“随手一句自然语言”相比，结构化 Prompt 更适合工程化复用。

## 6. 踩坑记录
- **常见问题**: Prompt 里没有明确输出格式。
- **错误现象**: 模型有时输出段落，有时输出 JSON，有时夹带解释。
- **定位方式**: 对比多轮结果，检查是不是任务、限制或示例描述不够清晰。
- **解决方案**: 明确输出结构、禁止项和必要字段，必要时补 few-shot 示例。

## 7. 面试高频 Q&A
### Q1: Prompt 设计最重要的能力是什么？
### A1:
最重要的是把模糊需求转成结构化任务，让模型知道“要做什么、依据什么、输出成什么样”。

### Q2: 为什么输出格式要写清楚？
### A2:
因为模型天然倾向生成自然语言，输出格式不明确时，结果很难稳定接入程序或工作流。
