## 1. 背景
- **问题场景**: 模型训练完成并不代表业务可用，真正的工程挑战是如何把模型稳定部署、观测和回滚。
- **学习目标**: 认识模型部署流水线的基础步骤，理解“训练完”和“上线可用”之间的差距。
- **前置知识**: 了解模型训练、推理服务和基础 API 概念。

## 2. 核心结论
- 模型部署不是单一步骤，而是一条包含打包、发布、验证、监控和回滚的流水线。
- 可重复部署和版本管理是 MLOps 的基础能力。
- 没有线上监控的部署，只能算把模型“放上去了”，不算真正可运营。
- 上线策略最好配合灰度、回滚和结果对比机制。

## 3. 原理拆解
- **关键概念**: 模型部署通常包含模型产物、推理服务、配置管理、流量接入和监控告警。
- **运行机制**: 训练完成后的模型被打包成可部署单元，通过服务平台上线，再由监控系统追踪线上表现。
- **图示说明**: 一条基本 MLOps 部署链路是“模型产出 -> 打包 -> 服务发布 -> 验证 -> 监控”。

```mermaid
flowchart LR
    A["训练完成模型"] --> B["版本化与打包"]
    B --> C["部署推理服务"]
    C --> D["健康检查 / 灰度验证"]
    D --> E["线上监控与回滚"]
```

### 4.3 如何验证
- 本地运行命令: 视部署平台而定，核心是验证健康检查、推理延迟和版本切换。
- 预期结果: 新版本可发布、可观测、可回滚。
- 失败时重点检查: 模型版本和代码版本是否匹配、依赖环境是否一致、监控指标是否接通。

## 5. 项目实践建议
- **适用场景**: 分类、召回、排序、生成式模型服务等线上推理场景。
- **不适用场景**: 只做离线实验、不存在实际线上调用链路的纯研究任务。
- **落地建议**: 把模型版本、特征版本、镜像版本和发布记录统一管理。
- **与其他方案对比**: 与手工上线相比，流水线化部署更适合长期迭代和线上治理。

## 6. 踩坑记录
- **常见问题**: 训练环境和线上服务环境不一致。
- **错误现象**: 本地验证正常，线上推理结果异常或服务无法启动。
- **定位方式**: 对齐依赖版本、特征处理逻辑、模型输入输出协议和容器环境。
- **解决方案**: 强化版本管理与部署前验证，不要把“训练成功”当作“可上线”。

## 7. 面试高频 Q&A
### Q1: 为什么模型部署不能只看服务是否启动成功？
### A1:
因为服务启动成功不代表预测质量、延迟、错误率和资源占用都达标，更不代表可以安全回滚。

### Q2: MLOps 里的部署和普通后端发布有什么共同点？
### A2:
共同点在于都需要版本管理、发布验证、监控和回滚；不同点在于模型服务还要考虑数据分布和效果漂移。
