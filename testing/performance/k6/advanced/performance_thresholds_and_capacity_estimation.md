## 1. 背景
- **问题场景**: 很多团队能跑压测，但无法把结果变成“是否达标”“还能承载多少量”的结论。
- **学习目标**: 建立阈值设计与容量预估的基本方法，避免性能测试只停留在图表层。
- **前置知识**: 了解 QPS、P95、错误率、CPU、内存和基本系统资源指标。

## 2. 核心结论
- 性能阈值必须和业务 SLA 或用户体验目标绑定，而不是随便拍脑袋。
- 容量预估不是只看“最高 QPS”，而是看在可接受延迟和错误率下能稳定承载多少流量。
- 一次压测的结果只能说明当次环境，不应该脱离环境条件做绝对结论。
- 阈值设计和容量预估都需要业务、研发、测试共同参与确认。

## 3. 原理拆解
- **关键概念**: 阈值用于判定“是否达标”，容量用于回答“最大可承载多少业务流量”。
- **运行机制**: 在不同负载阶段采集资源与响应指标，找出延迟、错误率或资源使用开始明显恶化的拐点。
- **图示说明**: 典型容量评估会寻找系统从线性增长到明显退化的临界区域。

```mermaid
flowchart LR
    A["定义 SLA"] --> B["执行分阶段压测"]
    B --> C["观察延迟 / 错误率 / 资源"]
    C --> D["识别性能拐点"]
    D --> E["给出容量建议"]
```

## 4. 实战步骤

### 4.1 环境准备
- 依赖版本: k6 CLI
- 安装命令: 参考官方文档安装

```bash
k6 version
```

### 4.2 核心代码

```javascript
export const options = {
  thresholds: {
    http_req_duration: ["p(95)<300", "p(99)<800"],
    http_req_failed: ["rate<0.005"],
  },
};
```

### 4.3 如何验证
- 本地运行命令: 在不同负载阶段执行同一脚本，横向比较结果。
- 预期结果: 能清晰回答“在给定阈值下，系统可稳定承载的负载范围”。
- 失败时重点检查: SLA 是否真实合理、环境是否一致、是否同时采集了应用与系统资源指标。

```bash
k6 run script.js
```

## 5. 项目实践建议
- **适用场景**: 上线前容量预估、扩容评估、性能回归门禁、资源预算评估。
- **不适用场景**: 环境高度不稳定或业务目标模糊时。
- **落地建议**: 报告里至少写清楚测试目标、环境、阈值、结论和扩容建议。
- **与其他方案对比**: 与只给原始图表相比，阈值和容量结论更利于业务和管理层理解与决策。

## 6. 踩坑记录
- **常见问题**: 只盯应用响应时间，不看资源瓶颈。
- **错误现象**: 明明压测结果变差，却说不清是数据库、网络、线程池还是压测机的问题。
- **定位方式**: 同步采集应用监控、系统资源、数据库指标和外部依赖指标。
- **解决方案**: 性能压测要和观测体系一起做，不要只跑脚本。

## 7. 面试高频 Q&A
### Q1: 容量预估为什么不能只看最高 QPS？
### A1:
因为最高 QPS 不代表可稳定承载。只有在延迟、错误率和资源都处于可接受范围时，那个负载才有业务意义。

### Q2: 为什么性能阈值要和 SLA 绑定？
### A2:
因为性能是否合格本质上是业务问题，不是工具问题。脱离业务体验目标设置阈值，结论通常没有参考价值。
